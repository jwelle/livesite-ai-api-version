# Live Site AI — Technical & Product Overview

A complete reference for the architecture, stack, data model, and behavior of the
Live Site AI application as it stands today.

---

## 1. Product Summary

**Live Site AI** is a SaaS tool for marketing agencies that resell GoHighLevel
(GHL). The core job-to-be-done:

> Give me, the agency owner, a one-click way to spin up a *personalized live
> demo page* for a prospect — with their own branding, an embedded GHL chat
> widget, an AI-generated voice-agent prompt tuned to their business, and a
> shareable public URL — so I can sell GHL voice/chat agents faster without
> having to hand-craft every demo.

### Primary user roles

| Role | Capabilities |
|---|---|
| **Agency user** (default) | Create/edit/delete their own demos, run AI enrichment, edit/regenerate voice-agent prompts, manage their own agency settings, view their own analytics. |
| **Admin** | All of the above + view all users/demos, suspend/reactivate users, promote/demote admins, view audit log, impersonate users (read-only — mutations blocked while impersonating). |
| **Public visitor** (prospect) | Visits `/demo/:slug` to see the embedded prospect site + chat widget + call-to-action. No authentication required. |

### Core flows

1. **Sign in** → Replit OIDC.
2. **Create a demo** → enter prospect's company name, website URL, voice-agent goal, tone, primary CTA.
3. **Run AI enrichment** → server calls OpenAI Responses API with `web_search` to research the business and generate a structured business profile + voice-agent package + a final voice-agent system prompt.
4. **Edit / refine the prompt** → user can hand-edit the working prompt, regenerate (replace or save-as-separate-version), or restore any prior version from history.
5. **Save Final Prompt** → status flips to `approved`; the final prompt is the "shipping" copy.
6. **Push to GHL** → currently a placeholder; real integration is the next major task.
7. **Share the demo URL** → `https://<host>/demo/<slug>` renders a public page with the prospect's website in an iframe and a GHL chat widget overlay.
8. **Track engagement** → views, call clicks, calendar clicks, and website-open clicks are logged per demo.

### Request lifecycle

1. Browser SPA loads from Vite (dev) or static build behind Express (prod).
2. Generated React Query hooks call `/api/...` with credentials (cookie-based session).
3. `authMiddleware` resolves the session ID from the cookie, refreshes the OIDC token if expired, loads the user from DB, and (if admin is impersonating) substitutes `req.user` with the target user while keeping `req.realUser` and `req.impersonation` set.
4. Route handlers validate input with the generated Zod schemas (`@workspace/api-zod`), perform Drizzle queries, and return JSON.
5. Mutating routes apply `blockDuringImpersonation` so admins reviewing as another user can never accidentally write data on their behalf.

### Codegen pipeline

```
lib/api-spec/openapi.yaml              ← edit this
        │
        │  pnpm --filter @workspace/api-spec run codegen
        ▼
lib/api-client-react/src/generated/    ← React Query hooks for the browser
lib/api-zod/src/generated/             ← Zod schemas for server route validation
```

The server imports request/response Zod schemas from `@workspace/api-zod`; the
browser imports typed hooks from `@workspace/api-client-react`. Both are
regenerated from the same `openapi.yaml`, so the contract cannot drift.

| `created_at` | indexed via `demo_id` |

## 6. AI Enrichment Pipeline

Implemented in `artifacts/api-server/src/services/enrichmentService.ts`.

### Inputs
- Business name, website URL, industry (optional).
- Voice agent goal (required).
- Desired tone, primary CTA, optional notes (all optional).
- Disclaimer (from agency settings, optional).

### How the call is made
- **OpenAI Responses API**, model defaults to `gpt-5-mini` (override with `OPENAI_ENRICHMENT_MODEL`).
- A fixed **system instruction** tells the model to act as a voice-agent architect, use web search as the source of truth, and never invent prices/credentials/hours/etc.
- The **user prompt** injects the per-demo inputs and demands a strict JSON shape.
- The `web_search` tool is enabled (with a fallback to `web_search_preview` for older API versions).

### Two layers of templating
1. **Model-side template (the system + user prompt)** — defines what facts to extract and what JSON shape to return.
2. **Local-side template (`buildFinalPrompt`)** — assembles the model's structured output into a fixed, opinionated Markdown voice-agent prompt with sections for Identity, Business Context, Conversation Goal, Tone, Opening, Qualification, Service Guidance, Objection Handling, Escalation, Booking, Boundaries, optional Compliance + Disclaimer, and Final Style. The skeleton is identical for every business; the *content* changes.

### Validation + safety nets
- Response text is JSON-extracted (handles code fences and prose surround).
- Validated with a permissive Zod schema (`enrichmentResponseSchema`) — extra fields tolerated, unknown shapes rejected.
- Each field has a defensive normalization (`asString`, `asStringArray`) with sensible fallbacks (e.g. `"unknown"` for missing facts, `"AI Receptionist"` for missing agent name).
- A `limitedResults` boolean is set when source notes are empty or the model returned mostly unknowns; the UI surfaces a yellow warning banner.

### Endpoints
| Endpoint | Behavior |
|---|---|
| `POST /api/demos/:id/enrich` | Full re-research. **Overwrites** `businessProfile`, `voiceAgentPackage`, `aiGeneratedPrompt`, `currentWorkingPrompt`. Status → `enriched`. Writes `ai_generated` history row. |
| `POST /api/demos/:id/regenerate` | Always re-calls OpenAI. Body: `{ mode: "replace" \| "separate" }`. **replace** = same overwrite as above + writes a `regenerated` history row. **separate** = only writes a `regenerated` history row with note `"Mode: separate — …"`; the demo's prompts are untouched. |
| `POST /api/enrich-business` | Standalone enrichment that doesn't persist (used for "preview enrichment" UX). |
| `GET /api/openai-status` | Returns `{ configured: boolean }`. Public — only reveals key presence, not the key. |

### Rate limiting
In-memory per-user limiter (5 requests / minute) gates `enrich` + `regenerate`.
This is sufficient for V1 single-instance deployments; a shared store (Redis or DB) would be needed for horizontal scaling.

### Why enrichment is slow
- A single Responses-API call performs multiple **live web searches** + page reads, then writes JSON. Real-world latency is **20–60 s**, occasionally higher for businesses with sparse online presence.
- We do not stream; the UI shows a spinner until the full response is returned.
- High-ROI improvements (planned, not built): streaming + progress states ("Searching the web…", "Drafting the agent…"), profile caching by website URL, and a two-step pipeline (research → instant local prompt assembly).

- **Exports** — "Export Markdown" + "Export JSON" buttons hit `getExportDemoMarkdownUrl` / `getExportDemoJsonUrl` for streaming downloads.

`src/pages/public-demo.tsx`:

## 8. Authentication & Authorization

### Auth flow (Replit OIDC)
1. User clicks **Log in** → frontend redirects to `/api/login`.
2. Server starts the OIDC code-flow with PKCE (`openid-client`), redirects to Replit's identity provider.
3. Provider redirects back to `/api/callback` with the auth code.
4. Server exchanges code → tokens, upserts the user in `users`, stores `{ access_token, refresh_token, expires_at, user }` in the `sessions` table, sets a session cookie.
5. On every subsequent request, `authMiddleware` reads the cookie, loads the session, refreshes the access token if expired, and populates `req.user`.

### Authorization layers
- **`requireAdmin`** — `req.realUser.role === "admin"` or 403.
- **`blockDuringImpersonation`** — if an admin is currently impersonating another user, all mutating endpoints return 403. Read endpoints behave normally so the admin can see what the user sees.
- **Per-resource ownership** — every demo / settings / export endpoint scopes its DB queries to `req.user.id` so a user cannot read or mutate another user's data.

### Account states
- `users.status` = `active` | `suspended`. Suspended users have their session cleared on the next request and receive 403.

### Admin console
- `/admin/users` — list, search, suspend, reactivate, promote, demote, impersonate.
- `/admin/demos` — view all demos across all agencies.
- `/admin/audit-log` — every admin action recorded (`admin_audit_log`).
- `/admin/impersonate/:userId` + `/admin/impersonate/exit` — toggles impersonation. Banner is rendered in the UI while active.

## 12. Security Posture

- OpenAI API key is **server-side only**; never returned to the browser.
- All AI inputs (especially `websiteUrl`) are validated as proper http(s) URLs both client- and server-side; the server runs `isValidHttpUrl` before invoking OpenAI.
- No app-side scraping of user-provided URLs (the model does the fetching via OpenAI's tool).
- Per-user 5/min rate limit on enrichment / regeneration.
- Per-resource ownership enforced on every demo / settings endpoint.
- Mutations blocked while an admin is impersonating another user.
- Suspended users have sessions invalidated on next request.
- pnpm `minimumReleaseAge: 1440` (24h) defends against fresh malicious npm publishes.
- `/openai-status` is public but only exposes a boolean, not the key.

## 13. Known Limitations & Future Work

### Push to GHL is a placeholder
`/api/demos/:id/push-ghl` and `services/ghlService.ts` are intentional stubs.
The intended V2 flow (per the planning conversation):
1. Agency adds a GHL Private Integration Token (or OAuth connection) in Settings.
2. Each demo is mapped to a `ghlVoiceAgentId`.
3. "Push to GHL" calls GHL's API to update the voice agent's system prompt with the demo's `finalSavedPrompt` (or `currentWorkingPrompt` fallback), pushing agent name/opening line/phone from the package.
4. On success: status flips to `pushed_to_ghl`, last-push timestamp recorded.
5. On failure: status preserved, GHL error surfaced verbatim.

### Enrichment latency
Single non-streaming Responses-API call with `web_search` is the dominant cost.
High-ROI improvements:
1. Streaming + progress state ("Searching the web…", "Reading {site}…", "Drafting…").
2. Two-step pipeline: research → fast local template assembly.
3. Cache `businessProfile` by `websiteUrl` so re-runs reuse research.
4. Cap web-search loop count.
5. Background-job pattern (kick off, poll, websocket).

### Other planned tasks
- **Conversational refine-with-AI** — side-panel chat that proposes prompt diffs the user can Apply / Reject / Apply-and-continue. Reuses the same OpenAI plumbing and `prompt_versions` table.
- **Per-industry templates** — vertical-specific prompt skeletons (dental vs HVAC vs law firm).
- **Editable final-prompt template in Settings** — per-agency house style.
- **Shared (multi-user) agencies** — currently `agency_settings` is keyed per-user; a true `agencies`/`organizations` table would let teammates share branding and defaults.
- **Durable rate limiting** — Redis or DB-backed for multi-instance deployments.
- **Schema migrations** — currently rely on `drizzle-kit push`; adding versioned migrations would harden production deploys.

## 14. Glossary

- **Demo** — A prospect-facing landing page generated for a single business. Has a unique slug, branding, an embedded chat widget, and a voice-agent prompt.
- **Business Profile** — Structured JSON describing a business (name, services, hours, etc.) produced by AI web research.
- **Voice Agent Package** — Structured JSON describing the *strategy* for an AI voice agent (name, tone, opening line, qualification questions, objection handlers, escalation rules).
- **AI-Generated Prompt** — The Markdown system prompt produced by `buildFinalPrompt` from the profile + package.
- **Working Prompt** — The user's currently-edited copy of the prompt.
- **Final Prompt** — The user-approved "shipping" copy.
- **Push to GHL** — The (currently placeholder) action that syncs the final prompt into a GoHighLevel voice agent.
- **Enrichment** — The end-to-end OpenAI research → structured output → final-prompt flow.
- **Regenerate** — Re-running enrichment with a choice between *replace* (overwrite the working prompt) or *separate* (save as a new history version only).
